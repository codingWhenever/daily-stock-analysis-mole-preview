# 选基决策系统数据基座、持仓导入与荐基规格

更新时间：2026-06-26

本文档用于规划 `/funds` 选基模块从“公开基金数据展示”升级到“可解释、可回测、可因人而异给出明确决策建议”的完整产品与工程规格。平台截图持仓导入只是其中一个数据来源层；完整系统的重心是夯实公开数据、持仓数据、用户画像、市场热点、回测校准、组合风险、荐基候选和决策动作，最终形成能持续迭代的基金决策工作台。

## 设计重心

导入助手不是产品中心，它只是解决“真实持仓怎么进系统”的入口。当前阶段的核心是：

- 建立足够可信的数据底座：公开基金数据、持仓、用户画像、市场环境、热点资金、费用和交易规则。
- 给出合理的持仓分析：仓位是否过高、重复暴露是否明显、组合承压能力如何、定投是否还适合继续。
- 给出明确的买卖决策：买入、定投、等待、持有、暂停、减仓、回避，每个动作都有证据、仓位和失效条件。
- 扩展荐基能力：不只分析用户已知基金，也能按行业、主题、市场热度和用户画像推荐值得关注或可直接行动的基金产品。
- 重构页面信息架构：让用户先看到“今天该看什么、该做什么”，再进入数据详情和导入维护。

## 总体产品闭环

```text
公开基金数据 + 市场热点 + 用户画像 + 真实持仓
  -> 数据质量与覆盖度检查
  -> 持仓聚合与风险暴露
  -> 基金质量、费用、回测、同类比较
  -> 热点行业与候选基金池
  -> 规则决策引擎
  -> AI 解释与反方审阅
  -> 市场级买入/卖出/资金流榜单
  -> 个人持仓动作与仓位建议
  -> 用户反馈与后验校准
```

## 一句话目标

给定基金名称或代码后，系统能够基于公开数据、用户画像、真实持仓、市场环境、热点资金、历史回测和风险承受能力，输出明确、可解释、可追踪的基金决策建议。

同时，系统应能在用户没有输入具体基金时，先基于公开市场和平台数据生成“今日荐基 / 热点行业候选 / 市场级买入榜 / 市场级卖出榜 / 资金流向榜”等列表，帮助用户发现值得进一步分析的具体基金产品。用户画像和持仓数据接入后，再生成个人持仓动作建议。

## 目标用户

- 已经在支付宝、京东金融、雪球 / 蛋卷等平台持有多只基金的个人投资者。
- 持仓数量可能达到几十只，手工逐条录入成本高。
- 目标不是看一份泛泛基金介绍，而是希望知道“这只基金现在适不适合买、该不该继续定投、已有仓位是否过重、是否要减仓或回避”。

## 产品假设

如果系统能把公开基金数据、真实平台持仓、用户风险画像和市场环境整合为一套结构化证据链，那么用户会更愿意把选基模块作为日常基金决策入口，因为它不只是展示数据，而是能给出有边界、有仓位、有失效条件的明确建议。

## 成功信号

定量信号：

- 用户能在 10 分钟内完成支付宝、京东金融、雪球三个平台的主要持仓导入。
- OCR 导入后，用户确认前字段识别准确率达到可用水平；基金代码、持有金额、份额、成本、收益等核心字段有字段级置信度。
- 至少 90% 已确认持仓能成功映射到基金代码，并参与全部视图合并、组合暴露和决策计算。
- 每条强决策建议必须有数据质量、回测或风险证据支撑，不输出无来源的买卖结论。
- 荐基列表能按行业 / 主题 / 基金类型给出候选池，并说明为什么推荐、适合谁、风险在哪里、何时失效。

定性信号：

- 用户能看懂为什么系统建议“买入 / 定投 / 等回撤 / 持有 / 暂停定投 / 减仓 / 回避”。
- 用户能清楚区分真实抓取、OCR 后用户确认、公开数据估算、缺失和过期数据。
- 用户可以手动修正持仓数据，并让修正后的数据成为唯一可信版本。

最低信号：

- 能从三平台截图导入并确认一份当前持仓。
- 每个平台默认成为一个独立账本。
- 全部视图按同一基金代码聚合展示，但不改变各平台账本原始归属。
- 决策层只使用用户确认后的持仓数据。

## 非目标

- 不接入支付宝、京东金融、雪球账号登录。
- 不抓取 Cookie、不模拟 App 私有接口、不绕过平台限制。
- 不把截图快照伪造成完整交易流水。
- 不长期保存用户上传的原图和原始 OCR 结果。
- 不让 LLM 自由发挥生成无证据的投资建议。
- 不承诺收益，不输出脱离用户画像和数据质量的确定性结论。

## 当前系统上下文

现有系统已经具备以下基础：

- `/portfolio` 侧已有账户、交易、现金流水、快照、风险、CSV 导入等组合能力。
- `/funds` 侧已在推进公募基金搜索、基金池、基金画像、同类分位、费率、回测、数据质量、账本账户化等能力。
- 现有执行计划见 `docs/funds-execution-plan.md`。本文档不替代执行计划，而是作为后续完整产品规格与工程拆解基线。

## 数据基座总览

完整决策系统至少需要六类数据。

| 层级 | 数据 | 主要用途 | 可信度要求 |
| --- | --- | --- | --- |
| 公开基金基础数据 | 基金代码、名称、类型、经理、规模、费率、净值、收益、同类排名 | 判断基金本身质量，生成候选池 | 必须标注来源、日期、缺口 |
| 基金组合暴露 | 行业配置、重仓股、持仓集中度、报告披露 | 判断风格、主题、集中度和持仓风险 | 季报/半年报天然滞后，必须显示 as_of |
| 市场与热点 | 指数估值、趋势、风格轮动、ETF 资金流、行业热度、宏观利率 | 判断当前是否适合追买、等待或减仓，生成荐基榜 | 热点和资金流必须有时间窗口 |
| 用户画像 | 风险等级、期限、回撤预算、流动性、投资经验、偏好限制 | 决策个性化和硬门槛 | 用户主动填写并可编辑 |
| 真实持仓 | 各平台基金、金额、份额、成本、收益、定投状态 | 仓位建议、重复暴露、组合风险 | 用户确认后才入决策 |
| 回测与后验 | NAV walk-forward、买入持有、定投、同类中位、费用拖累、历史命中 | 校准规则强弱 | 严禁未来数据，显示样本充分性 |
| 荐基候选池 | 行业 Top 产品、同类 Top 产品、低费率指数、优质主动、风险回避名单 | 主动推荐和关注列表 | 必须可追溯候选来源和筛选理由 |

## 持仓数据来源策略

持仓来源只是完整系统的一部分，但它是个性化决策的关键。

### 来源优先级

| 优先级 | 来源 | 定位 | 可用字段 |
| --- | --- | --- | --- |
| P0 主来源 | 支付宝、京东金融、雪球 / 蛋卷原 App 截图 | 当前真实平台持仓口径 | 基金、金额、份额、成本、收益、收益率、昨日收益、定投状态 |
| P0 校验源 | 基金E账户截图 | 全量核对、发现漏项 | 基金、份额、销售机构、账户归属 |
| P1 补充源 | CSV / Excel / 账单 / 交易流水 | 历史成本、真实买入卖出、赎回费复盘 | 交易日期、金额、份额、手续费、流水号 |
| P2 手工维护 | 用户手动新增/编辑 | 兜底修正 | 任意持仓字段 |
| 不采用 | 平台登录、Cookie、私有 API 抓取 | 暂不支持 | 账号安全和合规风险高 |

### 平台截图作为主来源的原因

基金E账户适合做全量持仓核对，但通常更接近“当前份额快照”。支付宝、京东金融、雪球原 App 截图包含更多平台口径字段，尤其是持仓成本、累计收益、收益率、昨日收益和定投状态，更适合支撑决策。

## 持仓导入助手规格

### 导入流程

```text
选择平台
  -> 上传多张截图
  -> 文件校验与预处理
  -> OCR 识别
  -> 平台模板解析
  -> 生成可编辑预览表
  -> 用户确认或修正
  -> 选择覆盖方式
  -> 写入该平台账本的最新可信持仓
  -> 删除临时图片和 OCR 草稿
```

### 上传限制

| 项目 | P0 限制 |
| --- | --- |
| 单次平台 | 一次只导入一个平台：支付宝 / 京东金融 / 雪球 |
| 图片数量 | 每批最多 30 张 |
| 单张大小 | 最大 10MB |
| 批次总大小 | 最大 150MB |
| 格式 | `png`、`jpg/jpeg`、`webp` |
| iPhone 16 Pro 普通截图 | 支持，典型竖屏截图约 1206 x 2622 px |
| HEIC / HEIF | P0 提示转 PNG/JPG，P1 再支持自动转换 |
| PDF 长截图 | P1 支持，P0 不作为主路径 |
| 尺寸 | 最小宽度 720px；超大图可自动压缩到 OCR 适配宽度 |
| 安全校验 | 检查真实文件头，拒绝 SVG、ZIP、可执行文件 |
| 隐私 | 图片和 OCR 草稿仅临时处理，确认或超时后删除 |

### OCR 引擎

P0 推荐 RapidOCR：

- 本地离线，免费开源。
- 默认支持中英文。
- 基于 ONNXRuntime，工程依赖比完整 PaddleOCR 和 EasyOCR 轻。
- 适合嵌入现有 Python / FastAPI 服务。

备选：

- PaddleOCR：高精度和表格文档能力更完整，但依赖更重，适合后续增强。
- ocrmac / Apple Vision：适合 macOS 本地增强，但不跨平台。
- AI 视觉模型：效果可能更强，但涉及成本和敏感图片上传，必须显式 opt-in。

必须有降级：

- OCR 模型首次下载或初始化失败时，页面允许用户粘贴 iPhone 实况文本识别出的文字。
- OCR 结果不直接入库，必须经过用户确认。

### 预览与编辑

预览页必须支持：

- 逐行编辑基金代码、基金名称、持有金额、持有份额、持仓成本、持有收益、收益率、昨日收益、定投状态。
- 忽略某一行。
- 合并疑似重复基金。
- 显示字段置信度和异常提示。
- 用户修正后，该字段来源标记为 `user_edited`。
- 保存后仍可在账本中继续编辑。

### 覆盖规则

用户确认后，导入结果成为该平台账本的最新 canonical 持仓数据。系统不长期保存原始图片和原始 OCR 文本。

提交时必须让用户选择：

| 模式 | 行为 |
| --- | --- |
| 全量覆盖该平台 | 该平台账本以本次确认结果为准；未出现的旧基金标记为移除 / 已不在当前持仓 |
| 仅更新出现项目 | 只更新本次截图里出现的基金；旧持仓继续保留 |

原则：

- 每个平台账本只保留一份当前可信持仓。
- 全部视图可以聚合同一基金，但不改动平台账本原始归属。
- 人工编辑后的数据优先级高于 OCR 自动识别。
- 再次导入时，如果 OCR 与人工编辑字段冲突，必须提示差异，不静默覆盖。

## Canonical 持仓数据模型

P0 不应把平台截图伪造成真实交易流水。建议新增基金持仓快照语义，或在基金账本内增加当前持仓表，而不是写入假的 `PortfolioTrade`。

建议字段：

```json
{
  "ledger_id": 1,
  "source_platform": "alipay",
  "fund_code": "005827",
  "fund_name": "易方达蓝筹精选混合",
  "market_value": 12345.67,
  "shares": 8888.88,
  "cost_basis": 11000.0,
  "avg_cost": 1.2375,
  "unrealized_pnl": 1345.67,
  "unrealized_pnl_pct": 0.1223,
  "yesterday_pnl": 12.34,
  "dca_status": "active",
  "data_as_of": "2026-06-26",
  "source_type": "platform_screenshot_user_confirmed",
  "source_confidence": "user_confirmed",
  "last_import_batch_id": "optional",
  "updated_by": "user_confirmed_import",
  "updated_at": "2026-06-26T10:30:00+08:00"
}
```

不保存：

- 原始截图长期文件。
- 原始 OCR 全量文本长期记录。
- 平台登录态、Cookie、账号密码。

可短期保存：

- `import_batch_id`。
- 解析错误摘要。
- 文件数量、平台、处理状态。
- 确认时间。
- 用于排障的低敏字段统计。

## 账本与全部视图

默认映射：

```text
支付宝 -> 支付宝账本
京东金融 -> 京东金融账本
雪球 / 蛋卷 -> 雪球账本
手动录入 -> 手动账本或用户指定账本
```

单账本视图：

- 只展示该平台自己的持仓。
- 不合并同一基金跨平台份额。
- 可查看该平台的成本、收益、定投状态。

全部视图：

- 按 `fund_code` 聚合跨平台持仓。
- 汇总金额、份额、成本、收益。
- 显示来源平台拆分。
- 决策层使用聚合后的总仓位，但解释里要说明各平台分布。

后续账本交互：

- 排序、重命名、删除 / 归档、主题色、批量移动、手动加减持仓，进入后续迭代。
- P0 只要求导入后能形成账本归属，并支持必要编辑。

## 用户画像问卷

用户画像不是展示标签，而是决策硬参数。

P0 问卷建议 8-10 题：

| 维度 | 问题 | 决策用途 |
| --- | --- | --- |
| 投资目标 | 增值、养老、教育、短期备用、资产配置 | 决定期限和收益目标 |
| 投资期限 | 3 个月、1 年、3 年、5 年以上 | 决定可承受波动品类 |
| 最大可承受回撤 | -3%、-8%、-15%、-25%、更高 | 作为仓位和动作硬约束 |
| 流动性需求 | 随时用钱、半年内用、长期不用 | 限制高波动或长封闭品种 |
| 投资经验 | 新手、买过货债基、买过股基 ETF、熟悉回撤 | 决定解释深度和动作强度 |
| 回撤行为 | 会赎回、暂停、持有、加仓 | 判断是否适合定投或主题基金 |
| 收入稳定性 | 稳定、不稳定、一次性资金 | 决定定投或分批策略 |
| 可投资资产区间 | 区间即可，不要求精确值 | 仓位比例和单基金上限 |
| 已有资产结构 | 存款、理财、基金、股票、债券、黄金等 | 避免重复暴露 |
| 偏好限制 | 不买海外、不买主题、不买新基金、不买高费率等 | 个性化过滤 |

结构化输出：

```json
{
  "risk_level": "C3",
  "drawdown_budget": 0.15,
  "investment_horizon": "3y_plus",
  "liquidity_need": "medium",
  "decision_style": "balanced",
  "loss_behavior": "hold_or_dca",
  "preferred_actions": ["dca", "pullback_entry"],
  "hard_constraints": [
    "avoid_high_concentration_theme_if_weight_over_25pct"
  ]
}
```

## 市场与热点数据基座

决策层不能只看基金历史表现，还要看当前市场环境。

P0 / P1 需要纳入：

- A 股主要指数趋势、估值分位、成交热度。
- 宽基、行业、主题 ETF 资金流。
- 风格轮动：成长 / 价值、大盘 / 小盘、红利 / 科技、消费 / 医药等。
- 基金重仓行业与当前热点的匹配度。
- 热点拥挤度：短期涨幅、资金流入强度、估值分位、新闻密度。
- 宏观环境：利率、汇率、海外市场、QDII 额度和节假日延迟。

市场热点只能作为 evidence，不替代净值、风险、回测和用户画像。

## 荐基功能规格

荐基功能用于解决“用户不知道该看哪只基金”的问题。它不是简单排行榜，而是先用公开市场、平台榜单、交易热度、资金流和基金质量生成市场级候选。前期榜单完全脱离用户画像和持仓，只做公开客观数据汇总；用户画像和真实持仓可用后，再叠加个人适配，生成因人而异的持仓动作建议。

### 荐基入口

至少包含以下榜单：

| 榜单 | 用途 |
| --- | --- |
| 今日荐基 | 基于公开市场数据、资金流、行业热度和基金质量筛出的具体产品候选 |
| 热点行业 Top10 | 当前市场热度、资金流和趋势较强的行业 / 主题 |
| 行业下 Top10 产品 | 每个热点行业下筛出最值得关注的基金 / ETF / 联接产品 |
| 市场级买入榜 | 基于公开买入笔数、买入金额、净流入、申购热度或平台买入热度生成，不依赖当前用户个人数据 |
| 市场级卖出榜 | 基于公开卖出笔数、卖出金额、净流出、赎回热度或平台卖出热度生成，不依赖当前用户个人数据 |
| 资金流向榜 | 将资金流入、流出、成交额和热度变化按时间窗口汇总 |
| 市场级观察榜 | 产品质量较好但短期过热、估值拥挤或数据不足，适合作为候选而非个人动作 |
| 个人持仓动作建议 | 在用户画像和已确认持仓同时存在后，生成买入、定投、等回撤、加仓、减仓、替换、回避等动作 |

### 候选池生成

候选池前期不来自用户已持仓基金，应覆盖公开可买基金、ETF、ETF 联接、主动权益、债基、货基、QDII、FOF 等类型。用户持仓只在个人持仓动作建议阶段介入。

候选来源：

- 公开基金收益榜、同类排名、规模和费率。
- 指数 / 行业 / 主题 ETF 及联接基金。
- 市场热点行业和资金流对应的产品。
- 低费率、规模适中、跟踪误差可控的指数产品。
- 长期回撤控制较好、同类分位稳定的主动产品。
- 后续在已确认持仓存在时，再补充用户当前持仓的相似替代产品。

### 行业 Top10 与产品 Top10

每日或盘中刷新时，先生成行业 / 主题 Top10，再为每个行业给出最多 Top10 产品。

行业评分建议：

| 维度 | 说明 |
| --- | --- |
| 趋势动量 | 指数近 5 / 20 / 60 日表现 |
| 资金流 | ETF 净申购、成交额、资金流入变化 |
| 估值温度 | 当前估值分位，避免只追高 |
| 热点强度 | 新闻、政策、公告、社媒或资金关注度 |
| 市场可交易性 | 是否开放申购、限购、暂停申购、QDII 额度、确认日 |
| 风险状态 | 波动、回撤、拥挤度和短期涨幅 |

产品评分建议：

| 维度 | 说明 |
| --- | --- |
| 质量 | 长短期收益、同类排名、基金经理、规模 |
| 成本 | 申购费、管理费、托管费、销售服务费、赎回费 |
| 风险 | 回撤、波动、恢复速度、下行捕获 |
| 市场热度 | 买入/卖出笔数、买入/卖出金额、净流入/净流出、成交额或平台热度 |
| 个人适配 | 后续个人动作阶段再判断是否符合用户风险等级、期限、流动性和偏好 |
| 仓位 | 后续个人动作阶段再判断用户是否已有同类产品，是否需要替换而非新增 |
| 可执行性 | 是否限购、暂停申购、QDII 额度、确认日 |
| 校准 | 历史策略命中、回测样本、数据质量 |

### 刷新频率

荐基榜可以动态变化，但必须避免为了“新鲜感”制造噪音。

| 场景 | 建议刷新 |
| --- | --- |
| 非交易日 | 每日一次 |
| 交易日盘前 | 开盘前生成初始榜 |
| 交易时段 | 最多每 3 小时刷新一次热点和资金流相关榜单 |
| 盘后 | 收盘后生成当日总结榜 |
| 用户画像或持仓变化 | 仅在已确认持仓存在时重算个人持仓动作建议 |

每次刷新要保留 `generated_at`、`market_phase`、`data_as_of`、`refresh_reason` 和 `ranking_version`。如果三小时内变化很大，页面应解释变化原因，例如资金流突变、行业回撤、持仓超限或数据源更新。

### 市场榜单与个人动作边界

市场级榜单和个人持仓动作必须分层：

- 市场级买入榜、卖出榜、资金流向榜可以先基于公开客观数据生成，完全不依赖用户画像和持仓。
- 市场级“买入”只表示公开数据中买入笔数、买入金额、净流入、申购热度或平台买入热度靠前，不代表系统建议某个用户买入。
- 市场级“卖出”只表示公开数据中卖出笔数、卖出金额、净流出、赎回热度或平台卖出热度靠前，不代表系统建议某个用户卖出。
- 榜单是推荐具体产品的前期数据实证；产品进入榜单后，仍需要查看费用、风险、限购、回测、市场状态等证据。
- 无用户画像或无已确认持仓时，不输出个人持仓动作建议。
- 用户风险等级不匹配时不能进入个人买入动作。
- 用户已有同类仓位过高时，个人动作应变为等待、替换或减仓。
- 热点过热时，个人动作可以是等回撤；但市场级榜单只显示过热、拥挤、资金流等客观证据。
- 费用、限购、赎回约束不利时，个人动作必须降级。

## 持仓分析规格

持仓分析是决策系统的另一条主线，不能只在用户输入单个基金代码时才工作。

组合层至少输出：

- 总资产和各平台账本分布。
- 基金类型占比：权益、债券、货币、QDII、FOF、指数、主题。
- 行业 / 主题集中度。
- 单基金仓位 TopN。
- 同一基金跨平台重复持仓。
- 同类产品过多或高度重叠。
- 当前收益、成本、亏损集中区间。
- 定投状态与是否需要暂停 / 继续 / 调整频率。
- 与用户画像的匹配度。
- 组合承压能力和极端回撤估算。

持仓层建议动作：

| 动作 | 场景 |
| --- | --- |
| 继续持有 | 基金质量和组合适配都可接受 |
| 继续定投 | 长期逻辑成立、估值不过热、用户风险预算充足 |
| 暂停定投 | 短期过热、同类仓位过高或用户现金压力增大 |
| 小幅减仓 | 单基金或单主题超过上限 |
| 替换观察 | 同类中存在更低费率、更稳或更适配产品 |
| 清理候选 | 数据质量差、长期落后、费用高或不适配用户 |

## 回测与校准

回测是决策可信度的核心，但必须避免未来函数。

P0：

- NAV walk-forward 回测。
- 买入持有基准。
- 定投基准。
- 同类中位基准。
- 参考指数基准。
- 费用假设与费率来源说明。
- 输出样本数量、窗口、交易次数、命中率、最大回撤、收益、超额收益。

P1：

- 按基金类型校准阈值。
- 按市场阶段校准：牛市、震荡、熊市、极端回撤。
- 按动作校准：买入、定投、暂停、减仓、回避。
- 输出 `calibration_status`: `insufficient` / `experimental` / `usable` / `strong`。

P2：

- 引入真实用户持仓后的组合级后验。
- 比较系统建议与用户实际持仓变化。
- 记录后续收益、回撤、持有体验和用户反馈。

## 抗风险能力评估

建议命名为“承压能力”或“抗风险观察”，避免误导为安全保证。

评分维度：

| 维度 | 指标 |
| --- | --- |
| 回撤韧性 | 最大回撤、极端阶段损失、同类回撤分位 |
| 波动稳定 | 年化波动、下行波动、收益分布偏斜 |
| 恢复速度 | 回撤恢复天数、恢复失败次数 |
| 同类防御 | 熊市 / 震荡市同类排名 |
| 集中风险 | 行业集中度、前十大持仓集中度、主题拥挤 |
| 组合适配 | 用户已有同类资产、单基金仓位、平台重复持仓 |
| 流动性与成本 | 申赎限制、赎回费、QDII 限购、确认日 |
| 数据质量 | 指标覆盖度、新鲜度、是否估算 |

输出：

```json
{
  "risk_resilience_score": 72,
  "label": "中高承压",
  "strengths": ["同类回撤控制较好", "恢复速度快于同类中位"],
  "weaknesses": ["行业集中度偏高", "热点拥挤度较高"],
  "blocking_issues": []
}
```

## 决策动作体系

目标是明确，而不是永远“谨慎关注”。但明确建议必须带条件、仓位、失效边界和证据。

建议动作：

| 动作 | 含义 |
| --- | --- |
| `strong_buy_dca` | 强烈适合定投或分批建仓 |
| `buy_small_position` | 可小仓位试探 |
| `wait_for_pullback` | 基金质量可以，但当前估值、涨幅或拥挤度不适合追 |
| `hold` | 已持有可继续 |
| `pause_dca` | 暂停定投，等待风险释放 |
| `trim_position` | 降低仓位 |
| `avoid` | 当前不适合，不建议买 |
| `research_only` | 数据不足，只能研究，不能下动作 |

每个动作必须输出：

```json
{
  "action": "wait_for_pullback",
  "conviction": "high",
  "position_advice": "已有仓位不超过权益资产的15%；无仓位等待回撤或分3-6期",
  "why": [
    "同类排名靠前",
    "回撤恢复能力较强",
    "当前主题拥挤度偏高"
  ],
  "invalid_if": [
    "近20日跌破趋势阈值",
    "基金经理变更",
    "行业集中度继续升高"
  ],
  "evidence": [
    "fund_profile",
    "backtest_calibration",
    "portfolio_exposure",
    "market_context"
  ]
}
```

### 决策硬门槛

- 数据质量阻断：只能 `research_only`。
- 用户风险画像低于基金风险：不得输出 `strong_buy_dca`。
- 现金安全垫不足：不得建议增加权益基金仓位。
- 同主题或同基金仓位已过高：不得继续加仓，只能持有、等待或减仓。
- 基金费用或流动性约束显著不利：必须在动作中体现成本。
- 回测校准不足：动作强度降级，不能显示为高 conviction。

### 建议强度

系统可以给出强建议，但强建议的定义是：

- 动作明确。
- 仓位明确。
- 触发条件明确。
- 失效条件明确。
- 证据明确。
- 不适用人群明确。

不是简单把语气写得更激进。

### 资金隐私闭眼模式

金额、总资产、持仓市值、成本、收益、收益率、平台分布等敏感字段默认进入“闭眼”模式。

默认行为：

- 首次进入 `/funds`、`/portfolio` 或相关决策页时，金额类字段默认显示为 `****`。
- 用户可手动点眼睛图标临时打开。
- 打开状态只在当前浏览会话有效；刷新或重新进入后默认继续闭眼。
- 截图导入预览页中，OCR 识别出的金额也默认遮蔽，编辑时再局部显示。
- 导出、分享、截图模式默认继续遮蔽金额。

可见范围：

| 数据 | 默认 |
| --- | --- |
| 基金名称 / 代码 | 显示 |
| 平台名称 | 显示 |
| 持有金额 / 总资产 / 成本 | 闭眼 |
| 收益金额 / 昨日收益 | 闭眼 |
| 收益率 | 可配置，默认闭眼 |
| 仓位百分比 | 可显示，因为决策需要，但可由用户改为闭眼 |
| 决策动作 | 显示 |

工程要求：

- 前端渲染层必须统一走金额脱敏 formatter。
- 日志、错误、通知摘要和 AI prompt 不应携带完整金额，除非明确需要且有低敏摘要替代。
- 决策层可以使用真实金额计算，但对外展示默认脱敏。

### 持仓页

导入助手步骤：

1. 选择平台。
2. 上传截图。
3. OCR 解析中。
4. 可编辑预览。
5. 选择全量覆盖或仅更新出现项目。
6. 写入账本并跳转到账本视图。

关键 UI 状态：

- OCR 模型准备中。
- 图片不符合限制。
- 字段低置信度。
- 疑似重复基金。
- 与已有人工编辑数据冲突。
- 导入完成但有未确认项。

持仓页还需要：

- 平台账本切换与全部视图。
- 全部视图下按同一基金合并，并显示平台拆分。
- 持仓动作标签：继续、定投、暂停、减仓、替换观察。
- 组合风险摘要固定在顶部。
- 每条持仓可进入“为什么这样建议”的证据抽屉。

### 荐基页

荐基页建议采用“榜单 + 行业矩阵 + 产品详情抽屉”。

核心区域：

- 热点行业 Top10：展示趋势、资金流、估值温度、拥挤风险。
- 行业下产品 Top10：每个行业可以展开产品列表。
- 市场级买入榜 / 卖出榜 / 资金流向榜：按公开市场或平台数据分组，不混成个人动作。
- 推荐候选：将市场级榜单中的具体产品沉淀为可分析、可关注、可加入基金池的候选。
- 个人持仓动作：在用户画像和已确认持仓可用后，再展示买入、定投、等回撤、减仓、替换等建议。
- 刷新说明：显示生成时间、市场阶段、刷新原因和数据质量。

产品卡片字段：

- 基金名称、代码、类型。
- 榜单类型、排名、数据口径和时间窗口。
- 市场级证据：买入/卖出笔数、买入/卖出金额、净流入/净流出、成交热度或平台热度。
- 推荐候选理由、主要风险、失效条件。
- 个人持仓动作、适合人群和仓位建议仅在用户画像与已确认持仓同时存在时展示。
- 推荐理由 Top3。
- 风险 Top3。
- 费用 / 限购 / 流动性摘要。
- 是否与当前持仓重叠。
- 加关注、加入基金池、生成决策报告。

### 基金详情页

基金详情页从“信息展示”改为“决策报告”：

- 第一块：当前动作、仓位建议、失效条件。
- 第二块：适合 / 不适合我的原因。
- 第三块：基金质量、同类位置、费用和交易规则。
- 第四块：市场与行业环境。
- 第五块：回测和校准状态。
- 第六块：数据质量与缺口。
- 第七块：AI 解释与反方观点。

## 隐私与安全

- 图片只做临时处理。
- 确认或取消后删除临时文件。
- 不保存平台账号、Cookie、登录态。
- 不上传图片给外部模型，除非用户明确选择 AI 视觉增强。
- 日志不得记录完整持仓明细和截图路径。
- API 返回错误不得暴露本地文件路径。
- 金额类展示默认进入闭眼模式，用户手动打开才显示。
- 通知、截图、分享和导出默认使用脱敏金额。
- AI 解释层默认接收低敏摘要，例如仓位百分比、风险等级、集中度，而不是完整平台金额明细。

## 验证计划

- 上传限制提示。
- 多截图预览。

- 冲突确认。
- 移动端不横向溢出。

## 分阶段落地

### P0：可信持仓导入与数据底座 MVP

- 三平台截图导入。
- RapidOCR 本地识别。
- 用户确认 / 编辑。
- 平台账本写入。
- 全部视图合并。
- 金额默认闭眼模式。
- 图片和 OCR 草稿临时处理。
- 未确认数据不参与决策。
- 公开基金基础数据、数据质量和市场上下文继续硬化。

### P1：用户画像、组合暴露与持仓分析

- 8-10 题画像问卷。
- 风险等级、回撤预算、期限、流动性、偏好限制。
- 聚合持仓的基金类型、行业、主题、平台分布和单基金仓位。
- 输出组合级风险摘要。
- 对每只已持仓基金输出继续、定投、暂停、减仓、替换观察等动作。

### P2：基金决策报告与荐基榜

- 单基金决策报告结合公开数据、回测、市场环境、用户画像和当前仓位。
- 输出明确动作、仓位、触发条件、失效条件和不适用人群。
- 记录决策版本和证据来源。
- 生成今日荐基、行业 Top10、行业下产品 Top10、市场级买入榜、市场级卖出榜、资金流向榜。
- 每个市场级候选都解释为什么上榜、数据口径、主要风险和何时失效；个人持仓动作另行解释适合谁、为什么现在不一定买、仓位和失效条件。

### P3：市场热点与校准增强

- ETF 资金流、行业热度、风格轮动、估值分位。
- 按基金类型和市场阶段校准策略。
- 热点拥挤度先进入市场级榜单解释和推荐候选排序；用户画像与已确认持仓可用后，再进入个人买入、等回撤或减仓动作判断。
- 交易时段最多每 3 小时刷新热点相关榜单，盘后生成当日榜单总结。

### P4：页面工作台与 AI 反馈闭环

- 完成 `/funds` 工作台 UI/UX 改版设计，覆盖今日决策、我的持仓、荐基、基金池、导入与数据、用户画像的信息架构和关键交互。
- UI 设计可先行，但工程实现需绑定数据与功能契约分步落地；页面必须明确区分已实接、计算中、待接入、需用户确认等状态，避免后续功能补齐时推翻结构。
- AI 负责把证据链解释清楚，补充反方理由和风险审阅。
- 用户反馈建议是否有用。
- 后验评估建议的实际表现。
- 逐步调参，但保留规则版本和证据可追踪。
