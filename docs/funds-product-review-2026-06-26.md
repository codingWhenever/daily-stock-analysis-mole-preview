# 选基产品目标、当前状态与路线评审稿

更新时间：2026-06-26

本文是 `/funds` 选基模块的最新评审版总览，整合以下文档：

- `docs/funds-decision-foundation-spec.md`
- `docs/funds-execution-plan.md`

用途：让产品、工程和后续 agent 在同一张图上评审“目标是什么、现在做到哪里、哪些还没做、下一步先做什么”。

## 1. 产品目标

最终目标不是做一个基金资料页，而是做一个“基金决策工作台”：

- 输入基金名称或代码时，给出公开数据、风险收益、费用、交易限制、同类比较、市场环境、回测校准和可解释动作。
- 用户没有输入具体基金时，系统能主动给出今日荐基、热点行业候选、市场级买入榜、市场级卖出榜、资金流向榜和产品推荐候选。
- 接入真实持仓后，能够结合用户画像、平台账本、仓位、成本、收益、流动性和风险承受能力，给出因人而异的持仓分析和买卖/定投/减仓建议。
- 建议可以明确、有力度，但必须带证据、仓位边界、触发条件、失效条件和不适用人群。

当前阶段的关键原则：

- 先证明数据，再验证策略，再输出决策，最后做 AI 解释和自动化荐基。
- 导入助手只是持仓数据来源，不是产品中心。
- 不把未来蓝图说成已完成。
- 不把公开数据、估算数据、缺失数据、过期数据混成一个“可用”标签。

本轮评审已确认的产品口径：

- 下一轮由系统自行在“已实现基础”和“最终目标”之间取舍：先收口 P0，再并行推进 P1 数据/回测、P2 市场上下文和 `/funds` 工作台 UI/UX 设计，工程落地按功能切片推进，避免 UI 与功能契约冲突后返工。
- 荐基候选不局限 ETF 或 ETF 联接，覆盖主动权益、指数、债券、货币、QDII、FOF 等可公开评价的产品；只是不同类型按数据质量分层准入。
- 买入榜和卖出榜前期完全脱离用户画像和持仓，只汇总公开客观市场/平台数据；买入表示公开数据里的买入笔数、买入金额、净流入或买入热度，卖出表示卖出笔数、卖出金额、净流出、赎回或卖出热度。
- 这些市场级榜单是推荐具体产品的前期数据实证，不是用户个人动作建议。
- 个人建议是另一层，只有在用户画像和已确认持仓都存在时才有实际意义；它再决定买入、定投、等回撤、加仓、减仓、替换、回避等动作倾向。
- 用户画像参考主流平台做轻量问卷，不做过重适当性模型；金额闭眼模式先只在 `/funds` 生效。
- 持仓导入主路径直接做 OCR 图片导入，RapidOCR + 用户确认/编辑为 P0 方案；粘贴文本只作为失败降级，不作为主方案。

当前 `/funds` 已进入“公开基金数据 + 本地基金池/账本 + 已确认持仓快照 + 市场级候选 + 个人动作 v2 预览”阶段。它仍不是完整投顾系统：个人动作只基于用户确认后的持仓、账本画像和已沉淀分析/市场上下文，不读取现金余额、完整交易流水或私有平台账号。

## 4. 优先级路线

### P0：当前成果收口与可信边界

目标：确保现有公开数据能力、回测和账本画像稳定可交接，不扩大范围。

任务：

- 对当前已提交切片做最终 review 和验证，保持未跟踪运行产物不进入功能提交。
- 确认 API、schema、版本号、文案和文档一致。
- 修掉“结构存在但能力未实接”的误导性表达。
- 补 `/funds` 桌面和移动端截图验收。
- 保持 `pytest 未安装时使用直接调用 test_*` 的验证边界说明。

验收：

- 当前能力能稳定运行。
- 文档、接口和 UI 不宣称未完成能力。
- 数据质量维度能清楚解释真实抓取、估算、缺失和过期。

### P1：数据底座与回测继续加固

目标：先把公开数据和策略验证做实，让后续决策和荐基有可信基础。

执行判断：P1 不必等所有市场数据完成才推进，但应与 P2 的最小市场上下文切片并行。优先补“能影响动作判断”的数据，如参考指数、ETF 资金流、行业趋势和估值温度，而不是继续堆静态展示字段。

任务：

- 扩大不同基金类型的样本和测试覆盖。
- 将回测校准结果持久化为快照，避免每次临时重算。
- 增加买入持有、定投、同类中位、参考指数等基准。
- 加强费用、交易限制、确认日、限购、赎回费阶梯对回测的影响。
- 针对货币、债券、指数、QDII 等类型补专项指标缺口。

验收：

- 能解释某类基金为什么只能 `experimental`，为什么不能自动改阈值。
- 每个 action 有样本数、命中率、收益/回撤统计和费用拖累。
- 继续保持禁止未来数据。

### P2：市场上下文实接

目标：把市场上下文从“结构/代理”推进到可用于决策的真实 evidence。

任务：

- 接入宽基、行业、主题指数趋势。
- 接入估值分位。
- 接入 ETF 资金流和成交热度。
- 接入风格轮动：成长/价值、大盘/小盘、红利/科技等。
- 为 QDII 增加海外市场、汇率、交易日延迟上下文。
- 为市场级买入榜、卖出榜和资金流向榜准备公开市场/平台数据底座，不混入用户个人持仓。

验收：

- `market_context` 能区分真实数据、代理数据和缺失数据。
- UI 不把缺口显示成已接入。
- 市场状态先影响市场级榜单证据、推荐候选排序和风险提示；在用户画像与已确认持仓可用后，再影响个人持仓动作。

### P3：资讯佐证实接

目标：让基金报告、重仓股公告、行业新闻和市场事件成为证据，而不是空结构。

任务：

- 复用现有 intelligence sources。
- 将 evidence 分为 `fund_report`、`holding_news`、`industry_news`、`market_event`。
- 每条 evidence 带来源、时间、关联标的/行业、正负面或风险标签。
- UI 中以“佐证 / 反证 / 风险提示”呈现，不堆新闻列表。

验收：

- 无资讯时 fail-open，不影响主流程。
- 有资讯时能解释其与基金、行业或重仓股的关系。
- 不用资讯替代净值、回测和风险指标。

### P4：账本交互增强

目标：让账本从后端分类变成可长期使用的基金工作台对象。

任务：

- 账本重命名、删除/归档、排序、主题色预设。
- 批量移动基金。
- 空账本和默认账本状态。
- 移动端按钮不挤压、不横向溢出。
- 展示账本画像：目标、风险、期限、调仓频率和备注。

验收：

- 用户能自然维护多个基金账本。
- 全部视图与单账本视图区分清楚。
- 不把账本画像误写成真实平台账户。

### P5：真实持仓导入与用户画像

目标：为个性化强决策打基础。

任务：

- 新增基金持仓快照语义，不伪造成交易流水。
- 三平台截图导入：支付宝、京东金融、雪球/蛋卷。
- 直接做 OCR 图片导入，RapidOCR 本地识别 + 用户确认/编辑。
- 基金E账户作为全量核对源。
- 用户画像问卷：参考主流平台做轻量版，覆盖风险等级、回撤预算、期限、流动性、经验、偏好。
- 金额默认闭眼模式，先只在 `/funds` 生效。

验收：

- 用户确认后的持仓才进入决策。
- 每个平台默认一个账本，全部视图只做同基金聚合展示。
- 金额类字段默认遮蔽，用户手动打开。

### P6：荐基与工作台 UI/UX 改版

目标：从“分析已知基金”升级为“主动发现适合用户的基金”。

任务：

- 继续增强今日荐基和推荐候选，补更多风险、费用、回测和市场状态证据。
- 继续增强热点行业 Top10 与行业产品 Top10，从当前代理版推进到行业资金、拥挤度和持仓穿透口径。
- 每个行业下的 Top 产品不局限 ETF/ETF 联接，也纳入主动权益、债券、货币、QDII、FOF 等公开可评价产品。
- 继续增强市场级买入榜、卖出榜、资金流入榜、资金流出榜和热度榜；买入侧已有平台公开热销榜，卖出侧仍需公开可核验的赎回/卖出热度来源。
- 将市场级榜单作为推荐具体产品的前期数据实证，继续进入今日荐基和行业产品候选。
- 只有在用户画像和已确认持仓都可用后，再从市场级榜单派生个人持仓动作建议。
- 交易时段热点榜最多每 3 小时刷新。
- 重新设计 `/funds` 工作台信息架构和关键交互，覆盖今日决策、我的持仓、荐基、基金池、导入与数据、用户画像。
- UI 设计可以先行，但页面状态必须跟数据契约同步标注 `已实接 / 计算中 / 待接入 / 需用户确认`，避免功能补齐时推翻结构。
- 工程落地按功能切片推进：先把会承载 P1/P2/P5/P6 的工作台骨架做对，再逐步接入真实数据和交互。

验收：

- 每个市场级推荐都有候选来源、数据口径、推荐理由、风险和失效条件；个人持仓动作再补适合人群、仓位和触发条件。
- 市场级榜单必须标注来源、数据时间、榜单口径和单位，例如买入笔数、买入金额、净流入、卖出笔数、卖出金额、净流出。
- 市场级买入榜、卖出榜不直接等同于个人买入/卖出建议。
- 无用户画像或无已确认持仓时，不输出个人持仓动作建议。
- 用户风险画像不匹配时不能进入个人买入动作。
- 当前持仓同类过高时，个人动作应降级为等待、替换或减仓。

### P7：AI 解释、反馈与后验闭环

目标：让 AI 做解释和复盘，不让 AI 绕过证据直接荐基。

任务：

- AI 解释证据链、反方观点、适用边界和失效条件。
- 用户反馈建议是否有用。
- 后验评估系统建议的实际表现。
- 根据后验结果校准规则和参数。

验收：

- AI 输出可追溯到结构化数据和规则信号。
- 每条强建议都有历史后验或明确样本不足提示。
- 用户反馈不直接改写策略，先进入分析和校准。

## 5. 本轮评审结论

本轮已确认以下执行口径：

1. 下一轮优先顺序由系统判断：P0 收口后，P1 数据底座/回测、P2 市场上下文和 `/funds` 工作台 UI/UX 设计并行推进。
2. 荐基候选不局限 ETF/ETF 联接，和持仓分析类似覆盖多类型基金；首批用数据质量和可解释性控制准入。
3. 买入榜、卖出榜前期是纯市场级公开数据榜单，可参考买入/卖出笔数、买入/卖出金额、净流入/净流出、成交热度、主流平台榜等，不依赖当前用户画像或持仓。
4. 市场级榜单是推荐具体产品的前期数据实证；个人买入、卖出、加仓、减仓建议另行生成，必须同时叠加用户画像和已确认持仓。
5. 用户画像参考主流平台做轻量版即可，先覆盖 8-10 个关键问题，不做过重模型。
6. 金额闭眼模式先只在 `/funds` 生效，后续是否扩展到 `/portfolio` 再评估。
7. 持仓导入直接推进 OCR 图片导入：RapidOCR 为主，用户确认/编辑为入库前置条件；粘贴文本只作为失败降级。
8. `/funds` UI 改版是重要关注点；先做精心设计的工作台信息架构和关键交互，再跟随数据底座、回测、市场上下文、榜单和导入流程分步落地。

## 6. 不应做的事

- 不要接支付宝、京东金融、雪球账号登录。
- 不要抓 Cookie 或模拟私有 App 接口。
- 不要把截图快照伪造成完整交易流水。
- 不要把荐基榜当成无条件买入推荐。
- 不要让 LLM 自由生成无证据买卖建议。
- 不要把当前仅第一版或部分实现的持仓导入、用户画像、金额闭眼、荐基 API、行业代理榜和买卖代理榜写成完整最终态。
